# ADR-0003: WebSocket 1本 + イベント受信箱 + catch-up

- Status: Accepted (2026-07-13)

## Context

要件は双方向: クライアント→サーバーの発話に加え、**エージェント発の能動通知**がある。
モバイルクライアントは頻繁に切断される。プッシュ通知基盤（FCM/APNs）は配信を保証しない。

## Decision

**配送チャネルを信頼しない。** 原則は「通知の真実はサーバーのDB、配送は best effort」。

1. エージェント発のイベント（応答・通知とも）は必ず先に `events` テーブルへ永続化し、単調増加IDを振る
2. 配送はその後。接続中の WebSocket に送る
3. クライアントは再接続時・起動時に `GET /events?after=<最終受信ID>` で差分を取り寄せる（catch-up）

転送は **WebSocket 1本の双方向**。REST(POST) + SSE の2チャネル構成と比較したが、
双方向が要件である以上、接続管理・認証・再接続を1本に集約できる WS が単純。

## プッシュ通知の段階設計

- **Phase 1（M4まで）**: フォアグラウンド WS のみ。非接続時のイベントは受信箱に溜まり、次回起動時に届く
- **Phase 2（必要になったら）**: FCM data message（本文なし、「新着あり」だけ）で端末を起こし、catch-up させる

受信箱が真実である限り、FCM は `Channel` 実装1個の追加であり、コア・プロトコルは無変更。
通知本文が Google のサーバーを経由しない副次効果もある。

## Consequences

- クライアントは最終受信IDを管理する責務を持つ
- events テーブルは追記専用で成長する。肥大化したら保持期間で削る（シングルユーザーなら当分先）

# ADR-0007: ジョブはプロセス内スケジューラ + 起動時再登録

- Status: Accepted (2026-07-14)

## Context

M4 の要件は「スキルが定期ジョブを登録でき、結果をエージェント発の通知として配信できる」こと。
再起動を跨いだスケジュール維持・多重実行防止・限定リトライ・実行履歴が求められる。

## Decision

**プロセス内スケジューラ（croner）+ 起動時再登録**方式を採る。

- スケジュール定義はコード（各スキルの `jobs`）にあり、起動時に毎回登録し直す。
  「次回実行時刻の永続化」はしない — 定義がコードにある以上、再登録で同じ結果になる
- cron 解釈は croner（依存ゼロ・タイムゾーン対応）。Asia/Tokyo 固定
- 多重実行防止は実行中ジョブ名の in-process セットで行う（シングルプロセス前提。
  ADR-0002 と同じ前提であり、これが崩れる時は Postgres 移行と同時に見直す）
- リトライは最大3回で打ち切り、全試行を `job_runs` テーブルに記録する

## 見送ったもの

- **ジョブキュー（BullMQ 等）**: Redis が必要になり VPS 1台構成の運用負担に見合わない
- **停止中に逃したジョブの追い実行**: 毎朝のサマリーは「逃したら次の朝」で困らない。
  必要になったら `job_runs` の最終成功時刻と比較して起動時に補完実行する拡張が可能

## Consequences

- ジョブの有効/無効は env（`DISABLED_JOBS`）で切り替え。実行履歴は SQLite で確認できる
- OSプッシュ（FCM）は引き続き ADR-0003 の Phase 2。通知は受信箱に永続化されるため、
  アプリ非起動時のジョブ通知も次回起動時の catch-up で必ず届く

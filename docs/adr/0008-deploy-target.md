# ADR-0008: 配備先は WebARENA Indigo 2GB、方式は systemd 直置き

- Status: Accepted (2026-07-16)

## 配備先: WebARENA Indigo 2GB (x86)

- **国内リージョンが体感に直結する**: チャットは WebSocket 往復 + ストリーミングであり、
  東京〜EU間の RTT（250ms前後）は毎ターンのレイテンシに乗る。国内なら一桁〜十数ms
- **2GB帯で国内最安級**（時間課金・月額上限あり・長期縛りなし）
- **Hetzner を選ばない理由**: 2026年の値上げに加え、安価な CAX（ARM）は EU リージョン限定。
  日本からの利用では優位性が消失した
- **対抗馬: KAGOYA CLOUD VPS 2GB**（月979円・日額課金・老舗でサポート厚め）。
  安定・サポート重視ならこちらだが、価格と課金の柔軟さで Indigo
- リスク受容: SLAなし・サポート最小限は個人用途として許容。
  配備完了直後に初期スナップショットを1枚取っておき、再構築コストを抑える

## 方式: systemd 直置き（Docker 化しない）

このサーバーの実体は「Bun バイナリ1個 + `bun install` + SQLite ファイル1個」。
コンテナが解決する問題（依存の分離・複数サービスの編成・チーム間の環境再現）が
この構成には存在せず、Docker 化はイメージビルドとレジストリ管理という新しい運用層を
足すだけになる。

- デプロイ単位は `git pull + bun install + systemctl restart`（deploy/deploy.sh）
- ロールバックは `git revert` して再デプロイ
- マイグレーションは起動時に自動適用（`store/db.ts` の設計をそのまま利用）
- 将来複数サービス化する場合に Dockerfile を足すのは容易（撤退可能な判断）

## バックアップ: 日次スナップショット（Litestream は見送り）

- 毎日 4:00 JST に `VACUUM INTO` でスナップショット、曜日名で7世代ローテーション。
  自前のジョブスケジューラ（ADR-0007）に `db-backup` ジョブとして登録する
  （外部 cron 不要・実行履歴が job_runs に残る・毎朝サマリーと同じ生存確認網に乗る）
- **Litestream（リアルタイムレプリケーション）は見送り**: S3互換ストレージの契約・運用が
  増える一方、失って痛いのは高々「最終スナップショット以降の会話数時間分」であり、
  個人運用の複雑さに見合わない

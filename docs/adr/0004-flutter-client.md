# ADR-0004: クライアントは Flutter + Riverpod

- Status: Accepted (2026-07-13)

## Decision

自作クライアントは **Flutter**（状態管理は Riverpod）で作る。

## Context / 理由

- **新技術の学習予算はサーバー側（Bun / エージェントループ）で使い切っている。**
  クライアントは既知の道具で速く作る側に置くのが、このプロジェクトの重心配分として正しい
- M1〜M4 の UI は最小限（メッセージリスト+入力欄+接続インジケータ）であり、
  ネイティブUIの表現力が活きる場面がない
- 開発中は macOS デスクトップ / シミュレータで動かし、実機配布は Phase 2（FCM導入）で考える

## Alternatives considered

- **PWA / Web**: 配布は最も楽だが、iOS の Web Push はインストール済みPWA限定で配信信頼性が低く、
  M4 以降の通知要件で確実に壁になる。将来要件から逆算して棄却
- **SwiftUI**: 個人用なので要件は絞れるが、学習コストの二重払いになる。
  `Channel` 抽象によりクライアント差し替えは自由なので、必要が生じた時に選び直せる
